# Бэкенд: как устроен пайплайн

Документ для команды. Описывает путь данных от сырого PDF до ответа в интерфейсе,
какие модули за что отвечают и какие решения приняты.

## Общая схема

```
 ЭТАП СБОРКИ (офлайн, запускается один раз)
 ┌──────────────────────────────────────────────────────────────────────┐
 │  data/raw (PDF, DOCX)                                                │
 │      │  ingest/parse.py  — парсинг, метаданные, нарезка на chunks    │
 │      ▼                                                               │
 │  documents.jsonl  — 1 строка = 1 документ                            │
 │      │  scripts/build.py — по каждому chunk:                         │
 │      │     core/extract.py  — LLM достаёт сущности/связи/числа       │
 │      │     core/graph_build.py — вливает их в граф (дедуп, источники)│
 │      ▼                                                               │
 │  data/processed/graph.pkl + documents.jsonl                          │
 └──────────────────────────────────────────────────────────────────────┘

 ЭТАП ОТВЕТА (онлайн, на каждый вопрос, ~2-5 сек)
 ┌──────────────────────────────────────────────────────────────────────┐
 │  вопрос → api/main.py POST /ask → core/rag.py:                       │
 │     1. retrieval.py   — BM25-поиск релевантных chunks                │
 │     2. llm.py         — генерация ответа ТОЛЬКО по найденному        │
 │     3. graph_store.py — подграф по теме вопроса                      │
 │     4. analysis.py    — пробелы + противоречия, флаги на узлы        │
 │  → один JSON: ответ + источники + граф + gaps + contradictions       │
 └──────────────────────────────────────────────────────────────────────┘
```

Ключевая идея: **ответ и граф — два независимых слоя над одними данными.**
Текст ответа модель пишет только по найденным фрагментам (никакой отсебятины),
а граф/аналитика считаются алгоритмически — их LLM не трогает, поэтому они объяснимы.

## Этап сборки

### 1. Парсинг (`ingest/parse.py`, владелец — Дубинин)
Вход: папка с PDF/DOCX. Выход: `documents.jsonl`, одна JSON-строка на документ:

```json
{"doc_id": "...", "source_path": "...", "title": "...", "year": 2023,
 "lang": "ru", "text": "полный текст", "chunks": ["кусок ~700 слов", "..."]}
```

Зачем какое поле:
- `doc_id` — паспорт документа, сквозной ключ по всей системе (источники, провенанс, /document);
- `chunks` — куски ~700 слов с перекрытием ~100: по ним идёт и поиск, и извлечение
  (документ целиком не влезает в контекстное окно модели);
- `title`, `year` — показываются пользователю в источниках, `year` — фильтр по времени;
- `lang` — рус/англ; `source_path` — прослеживаемость до файла.

### 2. Извлечение знаний (`core/extract.py`)
Каждый chunk уходит в LLM с промптом «верни строго JSON»: сущности (тип из онтологии —
Material, Process, Equipment, Property, Experiment, Publication, Expert, Facility; гео-метка
ru/foreign), связи (uses_material, produces_output, contradicts, ...) и числа с единицами
(«сульфаты ≤300 мг/л»). Ответ модели парсится защищённо: битый JSON не роняет сборку,
а возвращает пустой результат с логом.

### 3. Сборка графа (`core/graph_build.py`)
- **Дедупликация:** id узла строится из нормализованного имени — «Никель» из трёх статей
  становится одним узлом, к нему копятся источники.
- **Склейка свойств-синонимов:** «скорость циркуляции католита» и «скорость потока католита» —
  один узел (совпадение первого слова-величины + пересечение основ слов ≥ 50%).
  Без этого противоречия между источниками никогда не находились бы.
- **Провенанс:** каждый узел и ребро хранит doc_id всех документов, где встретился.
  На этом держатся источники, достоверность и поиск противоречий.
- **Числа** копятся списком измерений на узле-свойстве: {op, value, unit, source}.
- Сущность, упомянутая в связи, но пропущенная моделью в entities, получает узел-заглушку
  Unknown — связь ценнее чистоты типов; в анализ пробелов Unknown не попадает.

Хранилище — networkx MultiDiGraph за интерфейсом `GraphStore` (замена на Neo4j = одна новая
реализация интерфейса, остальной код не меняется).

## Этап ответа (`core/rag.py` — дирижёр)

1. **Поиск** (`core/retrieval.py`): BM25 по всем chunks, топ-6 со скором ≥ 2.0.
   Порог отсекает случайные совпадения по служебным словам: мусорный запрос («как приготовить
   борщ») даёт скор ~1.5 и честное «данных нет», осмысленный — от 3 и выше.
   Векторный поиск (bge-m3 / API-эмбеддинги) написан в `core/index.py`, но не подключён:
   на технических терминах BM25 работает надёжно и не требует GPU/API.
2. **Генерация** (`core/llm.py`): найденные фрагменты + вопрос → связный ответ.
   Системный промпт запрещает выдумывать и требует опираться на фрагменты.
   Бэкенд LLM выбирается конфигом: `LLM_BACKEND=yandex` (YandexGPT, выдан оргами)
   или `gigachat` — оба за одним интерфейсом `LLMClient`, ретраи на 429/5xx встроены.
3. **Подграф** (`graph_store.subgraph_for`): в вопросе находятся упомянутые узлы
   (сравнение по основам слов — переживает русские окончания), берётся их окружение
   в 1 шаг — это и есть картинка для vis-network.
4. **Аналитика** (`core/analysis.py`):
   - **Противоречия:** одно свойство, разные значения из РАЗНЫХ источников
     (диапазон 10–12 из одной статьи противоречием не считается) + явные рёбра contradicts.
   - **Пробелы:** материал и процесс считаются «изученными вместе», если делят источник;
     пробел = пары нет, но оба по отдельности изучены. Ранжируется по связности обоих.
   - Срез, который уходит в панели (топ-5 по теме вопроса), синхронно проставляется
     флагами на узлы подграфа — панель и подсветка графа не расходятся.

Если поиск ничего не нашёл — это не ошибка: HTTP 200, «в данных не найдено»,
confidence: low, пустые списки. Обработка пробелов — требование ТЗ.

## API (`api/main.py`)

| Эндпоинт | Что делает |
|---|---|
| `POST /ask` | главный: вопрос → ответ+источники+граф+gaps+contradictions |
| `GET /document/{doc_id}` | полный текст документа (переход из источников) |
| `GET /graph?limit=150` | обзорная «карта знаний»: самые связанные узлы |
| `GET /health` | статус: размер графа, число документов |

Ошибки: невалидный запрос → 422 (pydantic), LLM недоступен → 502 с текстом,
несуществующий документ → 404. Артефакты грузятся один раз при старте.

## Как запустить

```bash
pip install -r requirements.txt          # окружение
cp .env.example .env                     # + вписать ключи
python -m ingest.parse --input data/raw --output data/processed/documents.jsonl
python scripts/build.py --limit 50      # LLM-извлечение → граф
uvicorn api.main:app --reload            # API на :8000
```

Быстрая демо-проверка без парсинга: `make demo` (собирает граф из data/sample и поднимает API).

## Принятые решения (чтобы не переспрашивать)

| Вопрос | Решение | Почему |
|---|---|---|
| Обрезка текста для LLM | извлечение по chunks, не по документу | покрываем весь документ, а не начало |
| Узлы Unknown | оставляем | связь ценнее чистоты; в пробелы не попадают |
| Свойства-синонимы | склеиваем по основам слов | иначе противоречия не находятся |
| Поиск | BM25 с порогом 2.0, вектор в резерве | надёжно на терминах, ноль зависимостей |
| Источники в ответе | дедуп по doc_id | 6 кусков одной статьи ≠ 6 источников |
| «Нет данных» | HTTP 200 + честный ответ | это фича из ТЗ, не ошибка |
| Граф | networkx за интерфейсом | ноль инфраструктуры; Neo4j — одна реализация |
